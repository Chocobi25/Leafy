# REFACTORING

## 1. 리팩토링을 하는 이유
현재 프로젝트는 MVP 개발 단계에서 빠른 기능 구현을 우선으로 개발되었다.

그 결과:
- Service Layer 비대화
- 중복 조회 로직 증가
- DTO 생성 로직 중복
- Entity와 Response 스펙 불일치
- 트랜잭션 범위 과다
- N+1 문제 가능성

등의 문제가 존재한다.

서비스 운영 및 확장 전, 유지보수성과 성능 개선을 위해 리팩토링을 진행한다.

---

## 2. 현재 상황

- 현재 프로젝트는 리팩토링이 진행 중인 상태이며, 기존 구조와 신규 구조가 함께 공존하고 있다.
- 일부 도메인은 기존 Service 중심 구조를 유지하고 있으며, 일부 도메인은 Application / Find / Command Service 구조로 분리되었다.
- Redis 의존성은 점진적으로 제거 중이며, 관련 로직과 설정을 순차적으로 정리하고 있다.
- User, Post, Place, Trip 도메인은 우선적으로 리팩토링을 진행하였으며, 현재 리팩토링 이전 코드와 이후 코드가 일부 함께 존재한다.
- 리팩토링 과정에서는 기능 동작 유지와 점진적인 구조 개선을 우선하며, 한 번에 전체 구조를 변경하지 않는다.

---

## 3. 리팩토링 목표

### 1) Service 책임 분리 및 응답 속도 개선
- 기존의 거대한 서비스 레이어를 비즈니스 흐름 제어(Application Service), 데이터 조회 및 예외 처리(FindService), 상태 변경 및 DB 반영(CommandService)으로 책임을 엄격히 분리한다. 
- 외부 API 호출, 파일 I/O, 무거운 연산 로직은 @Transactional 범위 밖(Application Service 레벨)으로 전면 배치한다. 
- 순수 DB 작업만 트랜잭션 내부로 격리하여 DB 커넥션 점유 시간을 줄이고 커넥션 고갈 병목을 제거한다.

### 2) Query 최적화 및 N+1 문제 제거
- 복잡한 조회 조건, 동적 검색, 다중 필터링, 조회 전용 Projection 처리를 위해 QueryDSL을 도입한다.
- 단순 CRUD 및 간단한 단건 조회는 Spring Data JPA 메서드 조회를 유지하고, 복잡한 조회 로직에 한해 QueryDSL을 사용한다.
- 루프 내부에서 연관 엔티티를 반복 조회하는 패턴으로 인해 발생하는 N+1 문제를 전수 조사하고, fetch join, @EntityGraph, QueryDSL fetchJoin() 등을 활용하여 단일 쿼리 기반으로 최적화한다.
- 조회 전용 API는 Entity 전체 조회 대신 필요한 데이터만 조회하는 Projection 기반 조회를 우선 고려하여 불필요한 데이터 로딩을 최소화한다.
- 모든 FindService 조회 메서드에는 @Transactional(readOnly = true)를 적용하여 스냅샷 저장 및 Dirty Checking 오버헤드를 줄이고 조회 성능을 개선한다.

### 3) 중복 로직 제거 및 코드 가독성 향상
- 각 서비스 레이어마다 산재해 있던 Repository.findById().orElseThrow() 형태의 중복 조회 및 예외 처리 로직을 도메인별 FindService 내부로 이관하여 중복을 제거한다. 
- 상위 서비스 계층은 내부 구현을 구구절절 쓰지 않고, 잘 분리된 컴포넌트들을 조합해 "무엇을 하는지" 한눈에 읽히는 가독성 높은 코드로 리팩토링한다.

### 4) 코드 품질 및 유지보수성 향상
- 읽기 쉬운 코드를 우선하며, 복잡한 추상화보다 직관적인 흐름을 지향한다.
- 메서드는 가능한 짧게 유지하고 하나의 책임만 수행하도록 구성한다.
- 선언형 스타일을 지향하여 "어떻게"보다 "무엇을 하는지" 드러나는 코드를 작성한다.
- DTO 생성, Validation, 공통 계산 로직 등 반복되는 구현은 적절한 계층으로 분리하여 재사용성과 가독성을 높인다.
- Entity는 상태와 비즈니스 규칙을 직접 관리하도록 구성하고, 단순 데이터 컨테이너(Anemic Entity) 형태를 지양한다.

---

## 4. 리팩토링 제외 범위
현재 리팩토링에서는 다음 항목은 수행하지 않는다.
- MSA 전환
- 이벤트 기반 아키텍처 도입
- Redis / Kafka 등 신규 인프라 도입
- 과도한 Layer Architecture 적용

현재 구조를 유지한 상태에서 **유지보수성, 성능, 가독성 개선**을 우선 목표로 한다.

---

## 5. 리팩토링 진행 방식

- 기능 변경보다 구조 개선을 우선한다.
- 대규모 변경보다 작은 단위로 점진적으로 리팩토링한다.
- 기존 API 스펙은 최대한 유지한다.
- 동작 변경이 발생하는 경우 테스트 코드 또는 검증 과정을 함께 추가한다.
- 리팩토링 과정에서 발견된 기술 부채는 TODO 또는 문서로 기록한다.
- 특정 도메인 리팩토링 과정에서 다른 도메인 수정이 필요한 경우, 영향 범위를 최소화하고 필요한 작업은 TODO로 남긴다.
