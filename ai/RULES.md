# RULES.md

## 1. 작업 전 필수 절차
구현 전에 반드시 다음 단계를 거치며, **절대 코드를 바로 구현하지 않는다.**
1. 현재 구조 분석
2. 관련 클래스 탐색
3. 기존 패턴 확인
4. 영향 범위 설명
5. 구현 계획 먼저 제시 및 합의

---

## 2. 코드 작성 및 아키텍처 규칙

### 1) 함수 규칙
- 함수는 하나의 역할만 가지며 책임을 명확히 한다.
- 함수 길이는 가능하면 30줄 이하로 유지한다.
- 중첩 depth를 최소화한다.
- 긴 조건문은 의미 단위로 분리하여 메서드로 추출한다.

### 2) 네이밍 규칙
- 의미가 명확한 이름을 사용하며 축약어 사용을 최소화한다.
- boolean 변수 및 메서드는 `is` / `has` prefix를 사용한다.
- Repository 메서드에서만 `By` 키워드를 사용하고, 그 외 Service 및 비즈니스 메서드에서는 `By` 키워드 사용을 지양한다.
- 조회 메서드 접두어 규칙:
  - FindService의 모든 조회 메서드는 `find~`로 시작한다. (예: findUser)
  - Application Service의 모든 조회 메서드는 `get~`으로 시작한다. (예: getUser)

### 3) 에러 처리 규칙
- 모든 에러는 명확한 예외 메시지를 사용한다.
- 에러코드는 `global/exception/ErrorCode`를 상속해 도메인별로 정의하여 사용한다.
- 예상 가능한 비즈니스 에러는 Custom Exception을 정의하여 사용한다.
- 예외를 단순 묵인하는 try/catch 남용을 금지하며, 에러를 숨기거나 무시하지 않는다.

## 3. 금지 사항
- 프로젝트 전체 계층 구조 및 패키지 구조를 임의로 변경하지 않는다.
- 사용 목적이 불명확한 추상화(Abstraction) 및 인터페이스를 추가하지 않는다.
- 사전 논의 없이 외부 라이브러리(Dependency)를 추가하지 않는다.
- 현재 작업 범위와 무관한 파일 및 도메인을 수정하지 않는다.
- 외부 공공 API를 서비스 레이어에서 무분별하게 직접 호출하지 않는다.
- 재사용 목적이 불분명한 util class 생성을 지양한다.
- 단순 위임 역할만 수행하는 의미 없는 interface 분리를 지양한다.
- 과도한 mapper 분리 및 불필요한 DTO 변환 로직을 지양한다.
- 과도한 generic 사용으로 인해 가독성이 저하되는 구조를 지양한다.
- 복잡한 추상화보다 읽기 쉽고 유지보수하기 쉬운 구조를 우선한다.

## 4. 깃 규칙

### 1) 브랜치 규칙
브랜치는 다음 규칙을 따른다.
- feat/{작업내용}
- fix/{버그내용}
- refactor/{리팩토링내용}
- infra/{인프라작업}
- docs/{문서내용}
- chore/{기타작업}

예) refactor/user-service

### 2) 커밋 메세지 규칙

커밋 메시지는 다음 형식을 따른다.
- `{type}: {message}`
  - 사용 가능한 type: feat, fix, refactor, docs, chore, test, infra
  - 메세지는 한국어로 적는다.
- 예) refactor: User 서비스 책임 분리

### 3) 코드리뷰 규칙

- 리뷰 의견은 가능한 근거와 함께 작성한다.
- 단순 스타일 차이보다 구조, 가독성, 유지보수성, 성능 문제를 우선적으로 리뷰한다.
- 변경 범위가 큰 경우 작은 단위의 PR로 분리하는 것을 권장한다.
- 단순 취향보다는 프로젝트의 아키텍처 및 컨벤션 일관성을 우선한다.
- 애매한 구현은 추측하지 말고 의도를 질문한다.
